# Lyrics-bridge 插件接口文档

本文档描述 `Lyrics-bridge` 插件当前使用的 WebSocket 对接协议，供 EchoMusic-Lyrics-WinIsland 或其他外部软件实现服务端后与 EchoMusic 插件通信。

## 1. 连接方式

插件作为 WebSocket 客户端，外部软件需要作为 WebSocket 服务端。

默认连接地址：

```text
ws://127.0.0.1:17195
```

用户可以在 EchoMusic 的插件设置中修改服务端地址。

连接建立后，插件会主动发送：

1. 当前音乐数据同步事件 `MusicData`，仅当 EchoMusic 当前已加载到真实歌词时发送
2. 订阅事件 `subscribe`
3. 每 10 秒发送一次 `ping`

## 2. 通用消息格式

所有消息均为 JSON 字符串。

```json
{
  "type": "消息类型",
  "seq": 1,
  "payload": {}
}
```

字段说明：

| 字段 | 类型 | 方向 | 说明 |
| --- | --- | --- | --- |
| `type` | `string` | 双向 | 消息类型 |
| `seq` | `number` | 插件发出时存在 | 插件侧自增序号，用于调试或应答关联 |
| `payload` | `object` / `array` | 双向 | 消息载荷，不同 `type` 结构不同 |

说明：

- 插件不会强制要求服务端返回 `ack`
- 插件收到 `pong` 或 `ack` 会直接忽略
- 插件收到无法解析的 JSON 会忽略

## 3. 插件发送给服务端的消息

### 3.1 subscribe

连接建立后发送，用于声明插件希望接收的事件类别。

```json
{
  "type": "subscribe",
  "seq": 1,
  "payload": {
    "events": ["control", "state", "media"]
  }
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `payload.events` | `string[]` | 插件声明订阅的事件类别，目前固定为 `control`、`state`、`media` |

### 3.2 ping

插件连接成功后每 10 秒发送一次心跳。

```json
{
  "type": "ping"
}
```

服务端可以选择返回：

```json
{
  "type": "pong"
}
```

插件收到 `pong` 后不会执行额外逻辑。

### 3.3 MusicData

插件向服务端推送"当前曲目元数据 + 音乐封面 + 歌词（如有）"。无论当前是否有真实歌词，插件都会发送此事件。

```json
{
  "type": "MusicData",
  "seq": 2,
  "payload": {
    "Metadata": {
      "id": "123456",
      "title": "歌曲名",
      "artist": "歌手 A、歌手 B",
      "artists": ["歌手 A", "歌手 B"],
      "cover_base64": "/9j/4AAQSkZJRgABAQAAAQABAAD..."
    },
    "lyrics": [
      {
        "time_ms": 3430,
        "text": "第一句歌词"
      },
      {
        "time_ms": 15620,
        "text": "第二句歌词"
      }
    ]
  }
}
```

无歌词时的示例：

```json
{
  "type": "MusicData",
  "seq": 2,
  "payload": {
    "Metadata": {
      "id": "123456",
      "title": "歌曲名",
      "artist": "歌手 A、歌手 B",
      "artists": ["歌手 A", "歌手 B"],
      "cover_base64": "/9j/4AAQSkZJRgABAQAAAQABAAD..."
    },
    "lyrics": []
  }
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `payload.Metadata` | `object` | 当前歌词所属歌曲元数据，用于避免切歌或异步加载时产生歧义 |
| `payload.Metadata.id` | `string` | 歌曲标识，优先来自 EchoMusic 当前曲目的 `id`，兜底使用 `songId` 或 `hash` |
| `payload.Metadata.title` | `string` | 歌名，优先来自 `title`，兜底使用 `name` |
| `payload.Metadata.artist` | `string` | 歌手文本，多个歌手用 `、` 拼接 |
| `payload.Metadata.artists` | `string[]` | 歌手数组 |
| `payload.Metadata.cover_base64` | `string` | 音乐封面图片的 Base64 编码字符串；只包含 Base64 内容，不带 `data:image/...;base64,` 前缀；无封面或读取失败时为空字符串 |
| `payload.lyrics` | `array` | 歌词行数组，当前没有真实歌词时为空数组 |
| `payload.lyrics[].time_ms` | `number` | 歌词时间戳，单位毫秒 |
| `payload.lyrics[].text` | `string` | 歌词文本 |

重要行为：

- 无论是否有真实歌词，插件都会发送 `MusicData` 事件
- `payload.lyrics` 在无歌词时为空数组 `[]`
- 封面不存在或读取失败不会阻止发送，`payload.Metadata.cover_base64` 会为空字符串
- 歌词由异步加载时，插件先发送歌词为空数组的 `MusicData`，待歌词加载完成后再次发送完整 `MusicData`
- 歌词来源变化（如歌词重新加载、翻译/罗马音切换导致歌词数据变更）时也会发送一次完整 `MusicData`

触发时机：

| 时机 | 行为 |
| --- | --- |
| WebSocket 连接成功 | 发送一次 `MusicData` |
| EchoMusic 曲目切换 | 发送一次 `MusicData`（可能无歌词） |
| 歌词数据变更 | 同一歌曲的歌词数据发生变更（如异步加载完成、翻译/罗马音切换）时，发送一次完整 `MusicData` |

> 插件不自动重试推送歌词。若服务端收到的 `MusicData` 中 `lyrics` 为空数组，应自行通过 `request_MusicData` 指数退避轮询。

服务端重试机制：

服务端收到 `lyrics` 为空数组的 `MusicData` 后，应自行实现指数退避重试来获取歌词，而非依赖插件侧的定时推送：

```text
推荐退避策略：1s → 2s → 4s → 8s → 上限 30s
```

服务端在每个退避间隔向插件发送 `request_MusicData` 命令，插件在歌词就绪后会再次推送完整 `MusicData`。服务端收到带歌词的 `MusicData` 后应停止重试。

## 4. 控制命令（双向）

`command` 消息现在支持双向发送：服务端可以向插件发送命令控制 EchoMusic，插件也可以向服务端发送命令同步外部软件状态。

### 4.1 命令格式

```json
{
  "type": "command",
  "source": "server",
  "payload": {
    "action": "play",
    "data": {}
  }
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `source` | `string` | 命令发起方标识，`"plugin"` 或 `"server"` |
| `payload.action` | `string` | 控制动作 |
| `payload.data` | `object` | 动作参数，可选 |

### 4.2 服务端发送给插件的命令（source: "server"）

服务端可以通过以下 `action` 控制 EchoMusic：

| action | data | 说明 |
| --- | --- | --- |
| `request_MusicData` | 无 | 请求插件立即推送当前曲目的 `MusicData`；插件会尝试读取歌词并发送（可能仍为空数组），用于服务端指数退避重试 |
| `get_playback_state` | 无 | 请求插件立即回复当前播放状态；插件以 `source: "plugin" + action: "position"` 回复 |
| `get_software_info` | 无 | 请求插件回复软件信息；插件以 `source: "plugin" + action: "software_info"` 回复 |
| `upload_music_file` | `{ "upload_url": string }` | 请求插件将当前播放音频文件上传到指定地址；插件使用 HTTP POST + `Content-Type: audio/*` 发送二进制音频数据到 `{upload_url}/Music-file-upload`，并在请求头中附加歌曲元数据 |
| `show_notification` | `{ "title": string, "message": string }` | 在 EchoMusic 内显示提示消息 |
| `play` | 无 | 播放 |
| `pause` | 无 | 暂停 |
| `toggle_play` | 无 | 切换播放 / 暂停 |
| `next` | 无 | 下一首 |
| `prev` | 无 | 上一首 |
| `seek` | `{ "position_ms": number }` | 跳转到指定播放位置，单位毫秒；兼容 `time_ms`、`position`、`time`、`currentTime` 字段，其中不带 `_ms` 的字段按秒处理 |

#### seek 示例（服务端→插件）

```json
{
  "type": "command",
  "source": "server",
  "payload": {
    "action": "seek",
    "data": {
      "position_ms": 65000
    }
  }
}
```

插件会调用 EchoMusic 播放器跳转到 `65` 秒。

#### show_notification 示例

```json
{
  "type": "command",
  "source": "server",
  "payload": {
    "action": "show_notification",
    "data": {
      "title": "WinIsland",
      "message": "已连接"
    }
  }
}
```

插件会在 EchoMusic 内显示：

```text
WinIsland: 已连接
```

#### upload_music_file 示例与行为

服务端发送：

```json
{
  "type": "command",
  "source": "server",
  "payload": {
    "action": "upload_music_file",
    "data": {
      "upload_url": "http://192.168.1.100:8080"
    }
  }
}
```

插件收到后执行：

- 以 HTTP `POST` 请求发送到 `{upload_url}/Music-file-upload`
- `Content-Type: audio/*`
- 请求体为当前播放音频的二进制数据（从播放引擎或内存中直接获取，不产生额外网络请求）
- 附加自定义请求头携带歌曲元数据：

| Header | 值 | 说明 |
| --- | --- | --- |
| `X-Music-Id` | 当前曲目 `id` | 歌曲标识 |
| `X-Music-Title` | 当前曲目标题 | URL 编码后发送 |
| `X-Music-Artist` | 当前曲目歌手 | `、` 拼接，URL 编码后发送 |

插件不关注响应体内容，仅当 HTTP 状态码为 `200` 时视为上传成功，其余情况静默忽略。

### 4.3 插件发送给服务端的命令（source: "plugin"）

当 EchoMusic 的播放状态发生变化时，插件会主动向服务端发送以下命令：

| action | data | 触发时机 |
| --- | --- | --- |
| `play` | `{ "position_ms": number }` | 用户或系统切换到播放状态 |
| `pause` | `{ "position_ms": number }` | 用户或系统切换到暂停状态 |
| `toggle_play` | `{ "position_ms": number }` | 播放/暂停切换（仅在无法明确得知当前状态时发送） |
| `next` | 无 | 用户或系统切到下一首 |
| `prev` | 无 | 用户或系统切到上一首 |
| `seek` | `{ "position_ms": number, "cause"?: string }` | 用户拖动进度条、系统媒体控制等非自然播放进度变化 |
| `position` | `{ "position_ms": number, "duration_ms": number, "is_playing": boolean }` | 响应服务端 `get_playback_state` 命令，回复当前播放进度与状态 |
| `software_info` | `{ "name": string, "version": string, "author": string }` | 响应服务端 `get_software_info` 命令，回复软件信息 |

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `data.name` | `string` | 否（仅 software_info） | 软件名称，仅在 `action: "software_info"` 时存在 |
| `data.version` | `string` | 否（仅 software_info） | 软件版本号，仅在 `action: "software_info"` 时存在 |
| `data.author` | `string` | 否（仅 software_info） | 软件作者，仅在 `action: "software_info"` 时存在 |
| `data.position_ms` | `number` | 是（next/prev 除外） | 当前播放进度，单位毫秒 |
| `data.duration_ms` | `number` | 否（仅 position） | 当前曲目总时长，单位毫秒，仅在 `action: "position"` 时存在 |
| `data.is_playing` | `boolean` | 否（仅 position） | 当前是否处于播放状态，仅在 `action: "position"` 时存在 |
| `data.cause` | `string` | 否 | seek 的触发原因，取值 `"user"`、`"system"`、`"external"`，仅在 `action: "seek"` 时存在。服务端收到无 `cause` 的 seek 时应始终执行跳转 |

#### 自然推进不通知

播放进度的自然秒级推进不会触发 `seek` 通知。仅当插件检测到进度发生 **非自然跳转**（用户拖拽进度条、系统媒体控制跳转、外部 IPC 调用了 seek 等）时，才发送 `source: "plugin"` + `action: "seek"`。

#### seek 防闭环机制

1. 服务端发送 `source: "server" + action: "seek"` → 插件执行跳转
2. 插件**不**对此条命令产生的进度变化再次发送 `seek` 通知
3. 仅当 seek 的发起方 **不是** 服务端（用户拖进度条、系统媒体控制、EchoMusic 内部其他操作等）时，插件才发送 `source: "plugin" + action: "seek"` 通知给服务端
4. 服务端收到 `source: "plugin"` 的 `seek` 通知后，根据 `data.cause` 决定是否响应

#### seek 示例（插件→服务端）

```json
{
  "type": "command",
  "source": "plugin",
  "payload": {
    "action": "seek",
    "data": {
      "position_ms": 65000,
      "cause": "user"
    }
  }
}
```

#### software_info 示例（插件→服务端）

响应服务端 `get_software_info`：

```json
{
  "type": "command",
  "source": "plugin",
  "payload": {
    "action": "software_info",
    "data": {
      "name": "EchoMusic",
      "version": "2.5.0",
      "author": "hoowhoami"
    }
  }
}
```

#### position 示例（插件→服务端）

响应服务端 `get_playback_state`：

```json
{
  "type": "command",
  "source": "plugin",
  "payload": {
    "action": "position",
    "data": {
      "position_ms": 32000,
      "duration_ms": 245000,
      "is_playing": true
    }
  }
}
```

#### play / pause 示例（插件→服务端）

```json
{
  "type": "command",
  "source": "plugin",
  "payload": {
    "action": "play",
    "data": {
      "position_ms": 32000
    }
  }
}
```

### 4.4 pong

服务端可以响应插件的 `ping`。

```json
{
  "type": "pong"
}
```

插件收到后会忽略，不需要额外字段。

### 4.5 ack

服务端可以发送确认消息。

```json
{
  "type": "ack",
  "seq": 1
}
```

插件收到后会忽略，不需要额外字段。

### 4.6 state

插件当前只保留了 `state` 消息入口，暂未实现实际同步逻辑。

```json
{
  "type": "state",
  "payload": {
    "media": {}
  }
}
```

当前行为：

- 当 `type` 为 `state` 且 `payload.media` 存在时，插件不会报错
- 目前不会根据 `payload.media` 修改 EchoMusic 状态

## 5. 推荐服务端实现流程

1. 在本机启动 WebSocket 服务端，监听 `127.0.0.1:17195`
2. 等待 EchoMusic 插件连接
3. 收到 `subscribe` 后记录连接已就绪
4. 收到 `ping` 后可选返回 `pong`
5. 收到 `MusicData` 后按 `payload.Metadata` 确认歌曲，再处理 `payload.Metadata.cover_base64` 和 `payload.lyrics`
6. 收到 `source: "plugin"` 的 `command` 后同步外部软件状态
7. 如需控制播放，向插件发送 `source: "server" + command`

## 6. Node.js 服务端示例

```js
import { WebSocketServer } from 'ws'

const wss = new WebSocketServer({ host: '127.0.0.1', port: 17195 })

wss.on('connection', (socket) => {
  console.log('EchoMusic 插件已连接')

  socket.on('message', (raw) => {
    const msg = JSON.parse(raw.toString())
    console.log('收到消息:', msg)

    if (msg.type === 'ping') {
      socket.send(JSON.stringify({ type: 'pong' }))
      return
    }

    if (msg.type === 'MusicData') {
      const metadata = msg.payload.Metadata
      console.log('收到歌曲:', metadata.title, '-', metadata.artist)
      console.log('收到封面 Base64:', metadata.cover_base64 ? `${metadata.cover_base64.length} 字符` : '无')
      console.log('收到真实歌词:', msg.payload.lyrics)
    }
  })
})
```

## 7. 注意事项

- 插件是 WebSocket 客户端，外部软件必须先启动 WebSocket 服务端
- 默认地址是 `ws://127.0.0.1:17195`
- `MusicData` 的 `payload.Metadata.cover_base64` 是 Base64 文本，服务端需要自行解码后再作为图片使用
- 如果外部软件需要在切歌时清空旧歌词，应结合自身的播放状态来源处理，或等待下一次真实歌词覆盖
- 插件自动重连默认开启，断开后约 5 秒尝试重连
